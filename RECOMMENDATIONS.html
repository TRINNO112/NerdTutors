1 Making it work on low internet

The bandwidth problem is how images are sent, not their quality. Here's the current flow and where it breaks.

What happens today High impact

In ocr-scan.html, every selected photo is base64-encoded as-is (fileToBase64(), line 601) and all of them are sent together in a single request (line 682):

// ocr-scan.html — submit handler (line ~682)
body: JSON.stringify({
  mode: 'session-evaluate',
  images: uploadedFiles,        // ← ALL photos in one payload
  questions: selectedSession.questions,
  markingScheme: selectedSession.markingScheme,
  ...
})

Two problems on slow/flaky connections:

  • Base64 inflates each photo by ~33%. Three 4 MB photos → ~16 MB JSON.
  • ⚠️ Vercel serverless functions reject request bodies over 4.5 MB — so a normal multi-page sheet can fail before Gemini ever sees it. On slow internet the student waits minutes, then gets an error.
  • One dropped packet fails the entire submission — no retry.
  • The status text is static ("Uploading images…", line 671) so the student can't tell it's progressing and may refresh & double-submit.

Recommended changes (no quality loss)

#ChangeWhy it helps low internet
N1Upload images sequentially — one request per page, results mergedEach request stays well under 4.5 MB; a dropped page retries alone, not the whole sheet
N2Timeout + auto-retry (e.g. 2 retries, exponential backoff) via AbortControllerFlaky connections recover automatically instead of hard-failing
N3Real upload progress % using XMLHttpRequest.upload.onprogressStudent sees it's working → no panic refresh / duplicate submit
N4Resume-friendly drafts — you already cache the form; extend to remember which pages uploaded OKIf the connection dies mid-way, they don't restart from zero
N5Warn on very large / too many images at selection timeStops a guaranteed-to-fail 20 MB submission before it starts

N1 + N2 together solve the core low-internet failure with zero change to image quality.

2 Image quality — you were right, I walked it back

My first suggestion was aggressive compression. You correctly pointed out that low-quality images degrade OCR accuracy. Here's the corrected position.

❌ Dropped: aggressive compression

  • Downscaling to 1600px @ 0.7 quality can blur faint handwriting and hurt grading — exactly your concern.
  • Not worth the risk for an exam-grading tool where accuracy is everything.

✅ Kept: safe options only

  • Sequential upload (N1) is the real bandwidth fix — full quality preserved.
  • Optional safe-resize: only touch photos larger than ~2500px on the long edge, re-encode at 0.9 quality. Gemini internally tiles images to ~768–1536px anyway, so this trims wasted bytes without visible OCR loss.
  • This option is OFF by default — enable only if you want it.

👉 Recommendation: ship N1–N5, skip destructive compression. Decide separately whether you want the optional safe-resize toggle.

3 Making grading "moderate" (not harsh, not lenient)

The current rubric in ocr-evaluate.js (lines 172–181) stacks many mandatory heavy penalties. Combined, they guarantee harsh scores.

Why it's harsh today Calibration

"ZERO MARKS for off-topic truths… strict 0/3" · "deduct at least 1.5" (×3 different rules) · "deduct 0.5 for each spelling error" · "under 40 words → deduct 1.0"

A basically-correct answer with 3 spelling slips and slightly short prose can lose 3+ marks before it's even judged on content. That's the opposite of moderate.

Proposed "Moderate" rubric Your review

Off-topic answers still score 0 — kept per your instruction. What changes is the stacking of penalties on answers that are on-topic.

Current (harsh)

  • Off-topic-but-true → forced 0/3
  • ≥1.5 deduction for concept gap
  • ≥1.5 for not naming the term
  • ≥1.5 for wrong reasoning
  • 0.5 per spelling error (uncapped)
  • <40 words → −1.0 brevity cap

Proposed (moderate)

  • Off-topic → 0 (unchanged — your call ✔)
  • Partial credit in proportion to concept present (e.g. 2/3)
  • Wrong core concept → up to half marks off
  • Spelling/grammar → 0.5 max TOTAL, only if meaning is obscured
  • Brevity penalty only if answer is genuinely too thin to show understanding
  • MCQ wrong → 0; MCQ right → full (no partial)

Full draft prompt text:

⚠️ GRADING PHILOSOPHY — MODERATE (BALANCED):
Grade fairly and consistently. Reward demonstrated understanding of the
core concept; penalise real errors proportionally. Do NOT stack multiple
heavy penalties for the same mistake, and do NOT be lenient on factually
wrong or off-topic content.

MARK ALLOCATION:
- Full marks when the core concept is correct and adequately explained,
  even if wording is imperfect.
- Partial credit is EXPECTED: award marks in proportion to how much of the
  required concept is present (e.g. 2/3 for a mostly-right answer).
- MCQ: correct option = full, wrong option = 0. No partial credit, no guessing.
- OFF-TOPIC: if the answer does not address the question, score 0.
  If it addresses the question but includes some irrelevant material, grade
  the relevant part normally and note the irrelevance.
- Factual/conceptual errors: deduct in proportion to how central the error
  is (minor slip ≈ 0.5; core misconception ≈ up to half the marks).
- Brevity: only reduce marks if the answer is genuinely too thin to show
  understanding. A correct, concise answer is NOT penalised for length.
- Spelling/grammar: deduct at most 0.5 marks TOTAL per question, and only
  when errors obscure meaning or misuse a key technical term. Not per-error.
- Unattempted/blank = 0.

ACCURACY CHECK before returning JSON:
- Transcribe the student's answer faithfully.
- Each question's score must be between 0 and its maxMarks.
- "totalScore" must exactly equal the sum of individual scores.

Tune any number here (spelling cap, "half marks", brevity threshold) and I'll adjust before writing anything. This block would also be de-duplicated into one shared constant used by both session-evaluate and pdf-comparison modes, so future tuning happens in one place.

4 The Q1–Q8 = MCQ bug Confirmed bug

You flagged this and you're exactly right. Here is the actual code.

// api/ocr-evaluate.js  — lines 482–502
result.results.forEach(resObj => {
  const feedbackText = (resObj.feedback || "").toLowerCase();
  const isIncorrectText = feedbackText.includes("incorrect") ||
                          feedbackText.includes("wrong") ||
                          feedbackText.includes("should be") ||
                          feedbackText.includes("instead of") || ...;

  const isMCQ = Number(resObj.maxMarks) === 1 ||
                (resObj.questionNumber||"").toLowerCase().includes("mcq") ||
                (resObj.questionText||"").toLowerCase().includes("mcq") ||
                /^[qQ][1-8]\b/.test((resObj.questionNumber||"").trim());  // ← BUG

  if (isMCQ && isIncorrectText) {
    resObj.score = 0;   // force-zero, even for subjective answers
  }
});

Why it's wrong:

  • The regex /^[qQ][1-8]\b/ assumes every question numbered 1 through 8 is multiple-choice — a hardcoded assumption that breaks the moment Q1 is a 5-mark written answer.
  • It's combined with a text-keyword guess: if the AI's feedback simply contains "you should be more precise" or "this is incorrect in part", the whole answer is silently set to 0 — wiping out legitimately-earned partial marks.
  • Net effect: unpredictable, invisible zeros on subjective questions. Students can't tell why, and it reads as extreme harshness.

Recommended fix: stop guessing. Have the model return a structured field per question — e.g. questionType: "mcq" | "subjective" and for MCQs selectedOption / correctOption — then compare programmatically. Zero an MCQ only when the option letters genuinely don't match. No keyword scraping, no number-range guessing.
